# SPEC-GUIDANCE-001 수락 기준

## Module 1: 분쟁 케이스 탐지

### Scenario 1.1: 모호한 약관 조항 감지

```gherkin
Given 사용자가 인증된 상태이고
  And 채팅 세션이 활성화되어 있을 때
When 사용자가 "실손보험에서 도수치료가 보상이 되나요? 약관에 물리치료라고만 되어있는데"라고 질문하면
Then 시스템은 해당 약관 조항의 모호성을 감지하고
  And "도수치료"와 "물리치료"의 해석 차이를 ambiguous_clause로 태깅하고
  And 분쟁 분석 모드를 활성화해야 한다
```

### Scenario 1.2: 분쟁 의도 분류

```gherkin
Given IntentClassifier가 동작 중일 때
When 사용자가 "보험사에서 기왕증이라고 보상 거절했는데 이게 맞나요?"라고 질문하면
Then IntentClassifier는 dispute_guidance 의도를 반환하고
  And 분쟁 분석 파이프라인으로 라우팅해야 한다
```

### Scenario 1.3: 일반 보상 질의와 분쟁 안내 통합

```gherkin
Given 사용자가 보상 분석과 분쟁 안내 모두에 해당하는 질문을 할 때
When 사용자가 "교통사고로 입원했는데 보험사에서 인과관계가 부족하다고 거절했어요"라고 질문하면
Then 시스템은 보상 분석 결과(적용 가능한 담보)와
  And 분쟁 안내(인과관계 분쟁 분석)를 통합하여 응답해야 한다
```

### Scenario 1.4: 분쟁 미감지 케이스

```gherkin
Given 사용자가 명확한 보상 질의를 할 때
When 사용자가 "인공관절 수술비용이 실손보험에서 보상되나요?"라고 질문하면
Then 시스템은 분쟁 모드를 활성화하지 않고
  And 기존 보상 분석 파이프라인으로 응답해야 한다
```

## Module 2: 판례 검색 및 분석

### Scenario 2.1: 판례 검색 및 결과 제공

```gherkin
Given 분쟁 케이스가 탐지되었고
  And case_precedents 테이블에 관련 판례가 존재할 때
When 시스템이 판례 검색을 실행하면
Then 검색 결과에 사건번호, 판결일자, 판결 요지, 관련 약관 조항이 포함되어야 하고
  And 결과는 유사도 순으로 정렬되어야 한다
```

### Scenario 2.2: 상반된 판결 제시

```gherkin
Given "도수치료 보상 여부"에 대해 원고 승소 판례 3건과 피고 승소 판례 2건이 존재할 때
When 시스템이 판례 검색을 실행하면
Then 양쪽 판결을 모두 제시하고
  And 최신 판례가 상단에 정렬되어야 한다
```

### Scenario 2.3: 판례 부족 시 WebSearch 보강

```gherkin
Given 분쟁 케이스가 탐지되었고
  And case_precedents 테이블에 관련 판례가 3건 미만일 때
When 시스템이 판례 검색을 실행하면
Then WebSearch를 통해 최신 판례를 추가 검색하고
  And 검색 결과를 기존 판례와 통합하여 제시해야 한다
```

### Scenario 2.4: 출처 불명 판례 필터링

```gherkin
Given 판례 검색 결과에 출처 URL이 없는 항목이 포함될 때
When 시스템이 응답을 구성하면
Then 출처가 불명확한 판례는 결과에서 제외해야 한다
```

### Scenario 2.5: 판례 검색 성능

```gherkin
Given case_precedents 테이블에 10,000건의 판례가 저장되어 있을 때
When 시스템이 벡터 검색을 실행하면
Then 검색 결과가 200ms 이내에 반환되어야 한다
```

## Module 3: 확률 기반 성공 예측

### Scenario 3.1: 확률 점수 산출

```gherkin
Given 분쟁 분석이 완료되었고
  And 유사 판례가 5건 이상 존재할 때
When ProbabilityScorer가 확률을 계산하면
Then 0-100% 범위의 확률 점수가 산출되어야 하고
  And 유사 판례 승소율, 약관 해석 방향성, 금감원 분쟁조정 결과가 근거로 함께 제시되어야 한다
```

### Scenario 3.2: 판례 부족 시 확률 미제공

```gherkin
Given 분쟁 분석이 완료되었고
  And 유사 판례가 3건 미만일 때
When ProbabilityScorer가 확률을 계산하면
Then 확률 점수 대신 "판례 부족으로 정확한 예측이 어렵습니다" 메시지가 반환되어야 한다
```

### Scenario 3.3: 확률 표현의 법적 안전성

```gherkin
Given 확률 점수가 산출되었을 때
When 시스템이 사용자에게 확률을 제시하면
Then "약 65% 정도의 유사 사례에서 보상을 받았습니다"와 같은 참고 정보 형식으로 표현해야 하고
  And "65% 확률로 보상받을 수 있습니다"와 같은 확정적 표현을 사용하지 않아야 한다
```

## Module 4: 증거 전략 안내

### Scenario 4.1: 기왕증 분쟁 증거 안내

```gherkin
Given 분쟁 유형이 "pre_existing_condition"(기왕증)으로 분류되었을 때
When EvidenceAdvisor가 증거 전략을 생성하면
Then 다음 항목이 포함된 증거 체크리스트가 제공되어야 한다:
  | 증거 항목 | 설명 |
  | 시술 시점 증빙 | 최초 진단일과 시술일 간 시간 차이 증명 |
  | 의료 소견서 | 담당 의사의 기왕증 해당 여부 소견 |
  | 진료 기록 | 과거 진료 이력 (해당 질환 관련) |
  | 건강검진 결과 | 가입 전후 건강 상태 비교 |
```

### Scenario 4.2: 인과관계 분쟁 증거 안내

```gherkin
Given 분쟁 유형이 "causation"(인과관계)으로 분류되었을 때
When EvidenceAdvisor가 증거 전략을 생성하면
Then 사고와 상해 간 인과관계를 입증할 수 있는 증거 체크리스트가 제공되어야 한다:
  | 증거 항목 | 설명 |
  | 사고 경위서 | 사고 발생 경위 상세 기술 |
  | 진단서 | 사고 직후 의료기관 진단서 |
  | 의사 소견서 | 사고와 증상 간 인과관계에 대한 의학적 소견 |
  | 목격자 진술서 | 사고 목격자 확인서 (해당 시) |
```

### Scenario 4.3: 증거 수집 우선순위 안내

```gherkin
Given 증거 체크리스트가 생성되었을 때
When 시스템이 증거 전략을 사용자에게 제시하면
Then 각 증거 항목에 우선순위(높음/중간/낮음)가 표시되어야 하고
  And 시효 관련 주의사항이 있으면 함께 안내되어야 한다
```

## Module 5: 전문가 연계 추천

### Scenario 5.1: 전문가 상담 권장 (중간 확률)

```gherkin
Given 분쟁 성공 확률이 45%로 산출되었을 때
When EscalationAdvisor가 에스컬레이션 수준을 평가하면
Then "보험 전문가 상담을 권장합니다"라는 안내가 응답에 포함되어야 한다
```

### Scenario 5.2: 변호사 상담 강력 권장 (고액 분쟁)

```gherkin
Given 분쟁 금액이 1,500만원으로 확인되었을 때
When EscalationAdvisor가 에스컬레이션 수준을 평가하면
Then "보험 전문 변호사 상담을 권장합니다"라는 강한 안내가 제공되어야 한다
```

### Scenario 5.3: 공식 채널 정보 안내

```gherkin
Given 사용자가 전문가 연계를 요청했을 때
When 시스템이 공식 채널 정보를 제공하면
Then 다음 정보가 포함되어야 한다:
  | 채널 | 연락처 |
  | 금융감독원 민원센터 | 1332 |
  | 보험분쟁조정위원회 | 해당 연락처 |
  | 한국소비자원 | 1372 |
```

### Scenario 5.4: 법적 면책 고지 포함

```gherkin
Given 분쟁 분석 응답이 생성되었을 때
When 시스템이 최종 응답을 사용자에게 전송하면
Then 응답 말미에 다음 면책 고지가 포함되어야 한다:
  "본 안내는 교육적 목적의 정보 제공이며, 법적 자문이나 보험금 지급을 보장하는 것이 아닙니다."
  And "금융감독원 민원상담: 1332" 정보가 포함되어야 한다
```
