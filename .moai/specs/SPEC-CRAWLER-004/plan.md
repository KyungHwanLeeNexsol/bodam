# SPEC-CRAWLER-004: 구현 계획

## 마일스톤 개요

### Primary Goal: 흥국화재 크롤러 수정 (REQ-01)

**작업 내용:**
1. `crawl_nonlife_playwright.py`의 `heungkuk_fire` 메서드 분석
2. 수동 브라우저 접속으로 흥국화재 공시 페이지 구조 확인
3. 9개만 수집된 원인 진단 (페이지네이션, 카테고리 누락, 로딩 타이밍 등)
4. 판매중지 상품 접근 방식 확인 (탭/URL 파라미터)
5. 크롤러 코드 수정 및 전체 수집 실행
6. 수집 결과 검증 (PDF 수, 메타데이터 정합성)

**기대 결과:** 흥국화재 PDF 50개 이상 수집 (판매중지 포함)

### Secondary Goal: 5개 미실행 크롤러 실행 (REQ-02~06)

**작업 내용 (각 회사별):**
1. 해당 회사 웹사이트 접속 가능 여부 확인
2. 크롤러 dry-run 실행 (소수 상품만 테스트)
3. 실패 시 크롤러 코드 수정 (사이트 구조 변경 대응)
4. 판매중지 상품 접근 방식 확인 및 구현
5. 전체 수집 실행
6. 수집 결과 검증

**실행 순서 (독립적, 병렬 가능):**
- 한화손해보험 (hanwha_general)
- AXA손해보험 (axa_general)
- MG손해보험 (mg_insurance)
- NH농협손해보험 (nh_fire)
- 롯데손해보험 (lotte_insurance)

**기대 결과:** 각 회사 최소 1개 이상 PDF 수집, 가능하면 전체 질병/상해 상품 수집

### Tertiary Goal: 판매중지 검증 및 DB 인제스트 (REQ-07~09)

**작업 내용:**
1. 11개 활성 회사 전체에 대해 판매중지 상품 포함 여부 검증 스크립트 작성
2. 검증 실패 회사에 대해 판매중지 수집 보완
3. 수집 완료된 PDF 메타데이터를 Policy DB에 인제스트
4. 인제스트 결과 검증 (레코드 수, 중복 체크)
5. 전체 크롤링 결과 리포트 생성

**기대 결과:**
- 11개 회사 전체 판매중지 상품 포함 확인
- Policy DB에 새 레코드 추가
- 크롤링 리포트 JSON 파일 생성

### Optional Goal: 크롤러 안정성 개선

- 재시도 로직 강화 (지수 백오프 + jitter)
- 크롤링 중간 체크포인트 저장 (실패 시 이어서 수집)
- 회사별 크롤링 통계 대시보드 데이터 생성

## 기술적 접근 방식

### 크롤러 디버깅 전략

1. **Playwright Inspector 활용**: `PWDEBUG=1` 환경변수로 시각적 디버깅
2. **네트워크 로깅**: `page.on("response")` 이벤트로 API 호출 캡처
3. **스크린샷**: 각 단계에서 `page.screenshot()` 저장하여 상태 확인
4. **단계별 실행**: `--company` 옵션으로 개별 회사 크롤러 테스트

### 판매중지 상품 탐색 방법

1. 수동 브라우저 접속으로 사이트 구조 파악
2. DevTools Network 탭에서 API 엔드포인트 식별
3. 판매중지 필터/탭/URL 파라미터 확인
4. 크롤러 코드에 판매중지 수집 로직 추가

### 인제스트 파이프라인

```
PDF 수집 완료
  -> JSON 메타데이터 파싱
  -> SHA-256 해시로 중복 체크
  -> Policy DB upsert
  -> CrawlRun/CrawlResult 기록
  -> Celery 벡터 임베딩 태스크 트리거
```
