# SPEC-PDF-001 인수 기준: 온디맨드 보험 약관 PDF 분석 시스템

---

## Module 1: PDF 업로드 API

### AC-101: PDF 파일 업로드 성공

```gherkin
Given 인증된 사용자가 로그인한 상태이고
  And 유효한 PDF 파일(크기 10MB, 50페이지)을 준비했을 때
When 사용자가 POST /api/v1/pdf/upload 엔드포인트로 PDF 파일을 업로드하면
Then 시스템은 HTTP 201 상태 코드를 반환하고
  And 응답에 upload_id (UUID)가 포함되고
  And 응답에 파일명, 파일 크기, 상태(uploaded)가 포함되고
  And 파일이 uploads/pdf/{user_id}/{upload_id}.pdf 경로에 저장되고
  And pdf_uploads 테이블에 레코드가 생성된다
```

### AC-102: 유효하지 않은 파일 유형 거부

```gherkin
Given 인증된 사용자가 로그인한 상태이고
  And PDF가 아닌 파일(예: .exe를 .pdf로 확장자만 변경)을 준비했을 때
When 사용자가 POST /api/v1/pdf/upload 엔드포인트로 해당 파일을 업로드하면
Then 시스템은 HTTP 400 상태 코드를 반환하고
  And 응답에 "유효하지 않은 파일 형식입니다" 오류 메시지가 포함되고
  And 파일이 서버에 저장되지 않는다
```

### AC-103: 파일 크기 제한 초과

```gherkin
Given 인증된 사용자가 로그인한 상태이고
  And 51MB 크기의 PDF 파일을 준비했을 때
When 사용자가 POST /api/v1/pdf/upload 엔드포인트로 해당 파일을 업로드하면
Then 시스템은 HTTP 413 상태 코드를 반환하고
  And 응답에 "파일 크기가 50MB를 초과합니다" 오류 메시지가 포함되고
  And 파일이 서버에 저장되지 않는다
```

### AC-104: 사용자별 저장 용량 초과

```gherkin
Given 인증된 사용자가 이미 190MB의 PDF를 업로드한 상태이고
  And 15MB 크기의 새 PDF 파일을 준비했을 때
When 사용자가 POST /api/v1/pdf/upload 엔드포인트로 해당 파일을 업로드하면
Then 시스템은 HTTP 409 상태 코드를 반환하고
  And 응답에 "사용자 저장 용량(200MB)을 초과합니다. 기존 파일을 삭제해 주세요" 메시지가 포함된다
```

### AC-105: 비인증 사용자 접근 차단

```gherkin
Given 인증되지 않은 사용자(JWT 토큰 미포함)가 있을 때
When 해당 사용자가 POST /api/v1/pdf/upload 엔드포인트에 요청하면
Then 시스템은 HTTP 401 상태 코드를 반환하고
  And 응답에 "인증이 필요합니다" 오류 메시지가 포함된다
```

### AC-106: 파일명 살균 처리

```gherkin
Given 인증된 사용자가 파일명에 "../../../etc/passwd.pdf" 를 포함한 파일을 준비했을 때
When 사용자가 해당 파일을 업로드하면
Then 시스템은 파일명에서 경로 순회 문자를 제거하고
  And 저장된 파일명은 UUID 기반으로 생성되고
  And 원본 파일명은 살균(sanitize)된 형태로 데이터베이스에 저장된다
```

## Module 2: Gemini 2.0 Flash PDF 분석

### AC-201: 초기 보장 분석 성공

```gherkin
Given 인증된 사용자가 보험 약관 PDF를 업로드한 상태이고
  And upload_id가 유효할 때
When 사용자가 POST /api/v1/pdf/{upload_id}/analyze 엔드포인트로 분석을 요청하면
Then 시스템은 Gemini 2.0 Flash API를 통해 PDF를 분석하고
  And 응답에 다음 구조화된 결과가 포함된다:
    | 항목 | 설명 |
    | 보험 상품명 | 약관에서 추출한 상품명 |
    | 담보 목록 | 보장 항목 리스트 |
    | 보상 조건 | 각 담보별 보상 조건 |
    | 면책 사항 | 보상하지 않는 경우 |
    | 보상 한도 | 보상 금액 한도 |
  And pdf_analysis_sessions 테이블에 세션이 생성되고
  And initial_analysis 컬럼에 분석 결과가 JSON 형태로 저장된다
```

### AC-202: 자연어 질의 성공

```gherkin
Given 인증된 사용자가 PDF를 업로드하고 초기 분석이 완료된 상태이고
  And 활성 분석 세션이 존재할 때
When 사용자가 POST /api/v1/pdf/{upload_id}/query 로 "인공관절 수술은 보장이 되나요?"라는 질문을 전송하면
Then 시스템은 업로드된 PDF의 내용을 컨텍스트로 사용하여 답변을 생성하고
  And 답변에 약관 근거(해당 조항 번호 또는 내용 인용)가 포함되고
  And 대화 메시지가 pdf_analysis_messages 테이블에 저장되고
  And 토큰 사용량이 세션의 token_usage에 누적된다
```

### AC-203: SSE 스트리밍 응답

```gherkin
Given 인증된 사용자가 PDF에 대한 질의를 전송할 때
When 분석 결과가 긴 텍스트를 포함하면
Then 시스템은 Content-Type: text/event-stream 으로 응답하고
  And 분석 결과가 실시간으로 스트리밍되어 전달된다
```

### AC-204: Gemini API 오류 시 재시도

```gherkin
Given Gemini 2.0 Flash API가 일시적으로 오류를 반환하는 상태일 때
When 사용자가 PDF 분석을 요청하면
Then 시스템은 최대 3회 재시도하고 (지수 백오프 적용)
  And 3회 모두 실패하면 HTTP 503 상태 코드와 함께
  And "AI 분석 서비스가 일시적으로 불가합니다. 잠시 후 다시 시도해 주세요" 메시지를 반환한다
```

### AC-205: 토큰 사용량 추적

```gherkin
Given 사용자가 PDF 분석 또는 질의를 요청할 때
When Gemini API 호출이 완료되면
Then 시스템은 다음 토큰 정보를 기록한다:
  | 항목 | 설명 |
  | input_tokens | 입력 토큰 수 |
  | output_tokens | 출력 토큰 수 |
  | total_tokens | 총 토큰 수 |
  | estimated_cost_usd | 예상 USD 비용 |
  And 세션의 누적 토큰 사용량이 갱신된다
```

## Module 3: 분석 결과 캐싱

### AC-301: 동일 질의 캐시 적중

```gherkin
Given 사용자가 특정 PDF에 대해 "보장 범위가 어떻게 되나요?"라는 질문으로 분석을 완료한 상태일 때
When 동일한 사용자가 동일한 PDF에 대해 같은 질문을 다시 전송하면
Then 시스템은 캐시에서 결과를 반환하고
  And Gemini API를 호출하지 않고
  And 응답 시간이 1초 이내이다
```

### AC-302: 캐시 자동 만료

```gherkin
Given 분석 결과가 Redis에 캐싱된 상태이고
  And 24시간이 경과했을 때
When 동일한 질의가 들어오면
Then 시스템은 캐시 미스로 처리하고
  And Gemini API를 다시 호출하여 새로운 분석 결과를 생성한다
```

### AC-303: 분석 이력 조회

```gherkin
Given 사용자가 3개의 PDF 분석 세션을 완료한 상태일 때
When 사용자가 GET /api/v1/pdf/sessions 를 요청하면
Then 시스템은 3개의 세션 정보를 최신순으로 반환하고
  And 각 세션에 session_id, 파일명, 상태, 생성 시각, 마지막 활동 시각이 포함된다
```

## Module 4: 프론트엔드 PDF 업로드 UI

### AC-401: 드래그 앤 드롭 업로드

```gherkin
Given 인증된 사용자가 PDF 분석 페이지에 접속한 상태일 때
When 사용자가 PDF 파일을 드래그 앤 드롭 영역으로 끌어놓으면
Then 업로드 영역이 활성화 시각 효과를 표시하고
  And PDF 파일 업로드가 시작되고
  And 업로드 진행률이 백분율로 표시된다
```

### AC-402: 클라이언트 측 파일 크기 검증

```gherkin
Given 사용자가 PDF 분석 페이지에서 파일을 선택할 때
When 선택한 파일의 크기가 50MB를 초과하면
Then 프론트엔드는 업로드를 시작하지 않고
  And "파일 크기가 50MB를 초과합니다. 더 작은 파일을 선택해 주세요" 안내 메시지를 표시한다
```

### AC-403: 분석 결과 카드 표시

```gherkin
Given 사용자가 PDF 업로드 후 초기 분석이 완료되었을 때
When 분석 결과가 프론트엔드에 전달되면
Then 다음 항목이 구조화된 카드 형태로 표시된다:
  | 카드 항목 | 내용 |
  | 상품 개요 | 보험 상품명, 보험 종류 |
  | 담보 목록 | 보장 항목 리스트 (접기/펴기 가능) |
  | 보상 조건 | 각 담보별 보상 조건 요약 |
  | 면책 사항 | 보상하지 않는 경우 목록 |
  | 보상 한도 | 보상 금액 한도 정보 |
```

### AC-404: PDF 질의 채팅 인터페이스

```gherkin
Given 초기 분석이 완료된 PDF가 있을 때
When 사용자가 채팅 입력란에 질문을 입력하고 전송하면
Then 시스템은 스트리밍 방식으로 답변을 표시하고
  And 이전 대화 이력이 위에 표시되고
  And 새 메시지가 아래에 추가된다
```

### AC-405: 모바일 반응형

```gherkin
Given 사용자가 모바일 디바이스(화면 너비 375px)에서 접속할 때
When PDF 분석 페이지를 열면
Then 업로드 영역이 화면 너비에 맞게 조정되고
  And 분석 결과 카드가 세로 스크롤 형태로 표시되고
  And 채팅 인터페이스가 모바일에 최적화된 레이아웃으로 표시된다
```

## Module 5: 분석 세션 관리

### AC-501: 세션 자동 생성

```gherkin
Given 사용자가 PDF를 업로드하고 분석 요청이 성공적으로 처리될 때
When 분석이 시작되면
Then 시스템은 자동으로 PdfAnalysisSession을 생성하고
  And 세션 제목은 원본 파일명에서 파생되고
  And 세션 상태는 "active"로 설정되고
  And expires_at 은 현재 시각 + 24시간으로 설정된다
```

### AC-502: 세션 내 대화 컨텍스트 유지

```gherkin
Given 사용자가 활성 분석 세션에서 "실손보험 보장 범위가 어떻게 되나요?"를 질문한 상태이고
  And 시스템이 해당 질문에 답변한 상태일 때
When 사용자가 "그러면 치과 치료도 포함되나요?"라는 후속 질문을 하면
Then 시스템은 이전 질문과 답변을 컨텍스트에 포함하여
  And 실손보험 맥락에서 치과 치료 보장 여부를 답변한다
```

### AC-503: 세션 수 제한 적용

```gherkin
Given 사용자가 이미 5개의 활성 분석 세션을 보유한 상태일 때
When 사용자가 새 PDF를 업로드하고 분석을 요청하면
Then 시스템은 HTTP 429 상태 코드를 반환하고
  And "활성 분석 세션이 최대 5개입니다. 기존 세션을 삭제한 후 다시 시도해 주세요" 메시지를 반환한다
```

### AC-504: 세션 수동 삭제

```gherkin
Given 사용자가 활성 분석 세션 목록에서 특정 세션을 선택할 때
When 사용자가 DELETE /api/v1/pdf/sessions/{session_id} 를 요청하면
Then 시스템은 HTTP 200 상태 코드를 반환하고
  And 해당 세션의 상태가 "deleted"로 변경되고
  And 관련 PDF 파일이 파일시스템에서 삭제되고
  And 관련 캐시가 Redis에서 삭제되고
  And 관련 대화 메시지가 삭제된다
```

### AC-505: 다른 사용자 세션 접근 차단

```gherkin
Given 사용자 A가 생성한 분석 세션이 존재할 때
When 사용자 B가 해당 세션의 session_id로 조회를 요청하면
Then 시스템은 HTTP 404 상태 코드를 반환하고
  And 세션 정보가 노출되지 않는다
```

### AC-506: 세션 자동 만료

```gherkin
Given 분석 세션의 마지막 활동으로부터 24시간이 경과했을 때
When 세션 정리 스케줄 작업이 실행되면
Then 해당 세션의 상태가 "expired"로 변경되고
  And 관련 PDF 파일이 파일시스템에서 삭제되고
  And 관련 캐시가 Redis에서 삭제된다
```

## 성능 기준

| 지표 | 기준 | 측정 방법 |
|------|------|-----------|
| PDF 업로드 응답 시간 | < 3초 (50MB 기준) | API 응답 시간 측정 |
| 초기 보장 분석 | < 30초 (200페이지 기준) | Gemini API 호출 ~ 응답 완료 |
| 후속 질의 응답 | < 15초 | 질의 전송 ~ 스트리밍 시작 |
| 캐시 적중 응답 | < 1초 | Redis 캐시 조회 ~ 응답 |
| 동시 분석 세션 | >= 10 | 부하 테스트 |
| 프론트엔드 초기 로딩 | < 2초 | Lighthouse 측정 |

## 보안 기준

| 항목 | 기준 | 검증 방법 |
|------|------|-----------|
| 파일 유형 검증 | MIME 타입 + 매직 바이트 이중 검증 | 위장 파일 업로드 테스트 |
| 파일 크기 제한 | 50MB 초과 차단 | 대용량 파일 업로드 테스트 |
| 사용자 격리 | 다른 사용자 데이터 접근 불가 | 크로스 유저 접근 테스트 |
| 파일명 살균 | 경로 순회 공격 차단 | 악성 파일명 테스트 |
| 자동 삭제 | 24시간 후 파일/세션 만료 | TTL 검증 테스트 |
| 인증 필수 | 모든 엔드포인트 JWT 인증 | 비인증 접근 테스트 |
| 전송 암호화 | HTTPS 필수 | TLS 인증서 검증 |

## Definition of Done

- [ ] 모든 API 엔드포인트가 구현되고 OpenAPI 문서에 반영됨
- [ ] PDF 업로드 시 파일 유형, 크기, 매직 바이트 검증이 동작함
- [ ] Gemini 2.0 Flash API를 통한 PDF 분석이 정상 동작함
- [ ] SSE 스트리밍 응답이 프론트엔드에서 정상 표시됨
- [ ] 분석 세션 CRUD 기능이 동작함
